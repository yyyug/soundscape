**Status:** RESOLVED

- `osm_ids`: number[] (now at feature root, not nested in properties)

- `feature_value`: string (specific OSM tag value like "primary", "atm", "gd_intersection")

**Backend Tile Generation Pipeline:**
1. **Hot Cache (R2):** Pre-generated & cached tiles for frequently accessed areas
2. **PMTiles Archive (R2):** Pre-generated tiles for Hong Kong offline
3. **Live Neon/PostGIS:** `soundscape_tile(z,x,y)` function returns properly formatted features
4. **Overpass Fallback:** Live OSM data with SoundScape-compatible feature generation
5. **Empty Tile:** Fallback when no data available

## Data Workflow

### Hong Kong Offline Data (offline-hk/)

1. **Ingest Phase** (`ingest-hk.ps1`):
   - Downloads Hong Kong OSM PBF from Geofabrik (~27 MB)
   - Runs IMPOSM3 import to Docker PostgreSQL + PostGIS
   - Creates `osm_roads`, `osm_places`, `osm_entrances` tables
   - Applies `soundscape_tile(z,x,y)` function

2. **Generate Phase** (`generate-pmtiles.py`):
   - Queries all z16 tiles for Hong Kong bounding box
   - Calls `soundscape_tile(16, x, y)` for each tile
   - Generates SoundScape-compatible GeoJSON
   - Compresses with gzip
   - Builds PMTiles archive (~100-150 MB)

3. **Upload Phase** (`upload-to-r2.ps1`):
   - Pushes PMTiles to Cloudflare R2
   - Key: `pmtiles/hongkong-z16.pmtiles`
   - Sets Cache-Control headers

4. **Serving Phase** (worker.ts):
   - Worker fetches tiles from R2 PMTiles archive
   - Returns GeoJSON with all SoundScape-required fields
   - Tracks tile access for hot-cache promotion
   - Returns 304 on cache hit (ETag match)

## What Must Change If You Want AlaVia As Backend

Do not implement now (per request). Required changes would be:

1. Add SoundScape tile endpoint compatibility:
- Implement `/tiles/{z}/{x}/{y}.json` with SoundScape GeoJSON schema and ETag behavior.

2. Add schema adapter for intersections and route-place flows:
- Normalize AlaVia payload keys to SoundScape/EyePal expected keys.
- Include turn-road resolution, heading/bearing continuity, and route POI shape.

3. Add service-level caching semantics:
- ETag and TTL behavior consistent with SoundScape client expectations.

4. Add static asset hosting and flighting config support:
- Host experiments/control JSON and static learning assets.

5. Add operational parity:
- OSM ingestion/indexing pipeline and production-grade tile generation similar to SoundScape service stack.

## Recommendation Details

### Preferred path for parity and speed

Use SoundScape backend architecture for Maps core:
- Keep/stand up `svcs/data` tile service and OSM ingestion pipeline.
- Point iOS `ServiceModel.swift` service host to your deployment.

### Optional hybrid path

Use SoundScape backend for core map/tile APIs, and AlaVia only for optional value-added APIs (for example paid street view summaries), via a server-side adapter so the iOS app receives one stable schema.

## Decision

For this project stage, use SoundScape backend as primary for map features.

AlaVia can be reused only after adding a compatibility layer and data-contract extensions listed above.
